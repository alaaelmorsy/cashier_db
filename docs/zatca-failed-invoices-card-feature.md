# ميزة: بطاقة "فشل الإرسال" وشاشة الفواتير الفاشلة (ZATCA)

## الفكرة العامة

في شاشة الفواتير، بجانب مربع البحث، تُعرض 3 بطاقات إحصائية (فقط عندما يكون الربط
الإلكتروني/ZATCA مفعّلاً):

1. **تم الإرسال** (أخضر) — عدد الفواتير التي أُرسلت وقُبلت من هيئة الزكاة.
2. **في الانتظار** (أصفر) — عدد الفواتير التي لم تُرسل بعد.
3. **فشل الإرسال** (أحمر) — عدد الفواتير التي رُفضت من الهيئة أو فشل إرسالها.

عند الضغط على بطاقة **"فشل الإرسال"** تُفتح نافذة منبثقة (Modal) احترافية بها جدول
لكل الفواتير الفاشلة، مع أزرار إجراء لكل صف، وترقيم صفحات.

---

## 1. شرط الظهور (Gating)

- البطاقات الثلاث لا تظهر إلا إذا كان إعداد "تفعيل الربط الإلكتروني" مفعّلاً
  (Flag يشبه `zatca_enabled` في إعدادات النظام).
- الإحصائيات تُحسب فقط على الفواتير الخاضعة فعلياً للإرسال، أي التي تاريخ
  إنشائها **بعد** "تاريخ بدء الإرسال للهيئة" (حقل تاريخ اختياري يُضبط من شاشة
  إعدادات الربط الإلكتروني؛ الفواتير الأقدم منه مستثناة تماماً من الإحصاء لأنها
  خارج نطاق الإرسال أصلاً). إن لم يكن هذا التاريخ مضبوطاً، تُحسب كل الفواتير.

## 2. تصنيف حالة الفاتورة (منطق الحالة)

لكل فاتورة توجد حقول تخزّن نتيجة الإرسال، على سبيل المثال:

- `zatca_status`: قيمة نصية من `pending` / `submitted` / `accepted` / `rejected`
- `zatca_submitted`: تاريخ/وقت الإرسال (أو null إن لم تُرسل)
- `zatca_rejection_reason`: نص سبب الرفض (إن وُجد)
- `zatca_response`: نص/JSON كامل لرد الهيئة (إن وُجد)

قاعدة التصنيف (مهم أن تُطبَّق بنفس الترتيب — الرفض له الأولوية):

```
failed  = (zatca_status == 'rejected')
sent    = !failed AND (zatca_status IN ('submitted','accepted') OR zatca_submitted IS NOT NULL)
pending = !failed AND !sent   // أي كل ما تبقى
```

## 3. حساب الإحصائيات (Backend)

استعلام واحد يُرجع الأعداد الثلاثة، بشرط فلترة على "تاريخ بدء الإرسال" إن وُجد:

```sql
SELECT
  SUM(CASE WHEN zatca_status='rejected' THEN 1 ELSE 0 END) AS failed,
  SUM(CASE WHEN zatca_status<>'rejected'
           AND (zatca_status IN ('submitted','accepted') OR zatca_submitted IS NOT NULL)
      THEN 1 ELSE 0 END) AS sent,
  COUNT(*) AS total
FROM sales
WHERE (doc_type IS NULL OR doc_type='invoice')
  [AND DATE(created_at) >= :send_from_date]   -- فقط إن كان محدداً
```

ثم: `pending = total - sent - failed`.

يُفضَّل حساب هذه الإحصائيات ضمن استدعاء "تهيئة الشاشة" (init) الذي يجلب أيضاً
الإعدادات وعدد الفواتير الكلي، لتقليل عدد الطلبات (Round-trips)، ويُعاد الشكل:

```json
{
  "ok": true,
  "settings": { "zatca_enabled": true, ... },
  "total_invoices": 1234,
  "zatca_stats": { "sent": 10, "pending": 3, "failed": 3 }
}
```

## 4. واجهة البطاقات (Frontend)

- 3 عناصر `div` بجانب مربع البحث، مخفية بشكل افتراضي (`display:none` أو class
  `hidden`)، تظهر فقط عند توفر `zatca_enabled=true` و`zatca_stats` في رد التهيئة.
- كل بطاقة: رقم كبير + تسمية صغيرة تحتها، بألوان مميزة (أخضر/أصفر/أحمر) وحدود
  خفيفة من نفس اللون.
- بطاقة "فشل الإرسال" فقط قابلة للنقر (`cursor:pointer` + hover) وتفتح المودال.

## 5. شاشة/مودال الفواتير الفاشلة

### الشكل والتصميم
- Overlay كامل الشاشة (خلفية داكنة شفافة + blur)، وصندوق أبيض بعرض أقصى ~1100px
  في المنتصف، بحد أقصى 90% من ارتفاع الشاشة.
- **رأس (Header)**: خلفية متدرجة حمراء، عنوان "الفواتير التي فشل إرسالها لهيئة
  الزكاة"، وعدّاد "العدد: N" (العدد الإجمالي وليس عدد الصفحة الحالية)، وزر إغلاق
  (✕) في الزاوية.
- **المحتوى**: حالة تحميل (Spinner)، حالة فارغة ("لا توجد فواتير فاشلة الإرسال
  🎉")، أو جدول.
- **أعمدة الجدول**:
  1. رقم الفاتورة
  2. جوال العميل (وليس اسمه)
  3. التاريخ **والوقت** معاً (وليس التاريخ فقط)
  4. طريقة الدفع (كاش/شبكة/آجل/مختلط/تمارا/تابي) بدل عرض سبب الفشل كعمود
     منفصل (سبب الفشل صار متاحاً فقط عبر زر "الرد")
  5. الإجراءات: 3 أزرار لكل صف:
     - **عرض الفاتورة** (أزرق) — يفتح نفس نافذة الطباعة المستخدمة في الجدول
       الرئيسي (تمرّر `id`, `pay`, `cash`, `cashier`, `refresh=1`, `reprint=1`).
     - **📤 إرسال يدوي** (أخضر) — يعيد محاولة إرسال هذه الفاتورة تحديداً؛ بعد
       الإرسال يُعاد تحميل **نفس الصفحة الحالية** من الجدول (وليس إعادة فتح
       المودال من الصفحة 1) حتى لا يفقد المستخدم مكانه.
     - **📄 الرد** (بنفسجي) — يظهر فقط إن وُجد `zatca_response` أو
       `zatca_rejection_reason`؛ يفتح نافذة صغيرة منفصلة تعرض الرد/السبب
       كنص/JSON منسّق.
- **تذييل (Footer)**: عناصر ترقيم الصفحات على اليمين/اليسار وزر "إغلاق" على
  الجانب الآخر.
- الإغلاق يتم بثلاث طرق: زر ✕ في الرأس، زر "إغلاق" في التذييل، أو الضغط خارج
  الصندوق (على الـ overlay نفسه).

### ملاحظة تقنية مهمة (طبقات Z-index)
عند فتح مودال "الرد" فوق مودال "الفواتير الفاشلة"، يجب رفع z-index الخاص بمودال
الرد فوق مودال القائمة (مثلاً القائمة `z-index:1000` والرد `z-index:1100` عند
عرضه)، وإلا سيظهر مودال الرد **خلف** مودال القائمة وهي مشكلة حقيقية واجهناها.

### الترقيم (Pagination)
- حجم الصفحة: 20 عنصر.
- عناصر التحكم: ⏮️ (أول صفحة) / ◀️ (السابقة) / نص "صفحة X من Y" / ▶️ (التالية)
  / ⏭️ (آخر صفحة)، مع تعطيل الأزرار عند الحدود.
- تُخفى عناصر الترقيم بالكامل إذا كانت الصفحات = 1.
- الترقيم من جهة الخادم (Server-side): يُطلب `page` و`pageSize` من الـ API، ولا
  تُجلب كل السجلات دفعة واحدة.

## 6. الفلترة على مستوى الـ API (Backend)

أضِف باراميتر فلترة `zatca_status` إلى نقطة استعلام الفواتير الموجودة (بدلاً من
إنشاء endpoint جديد)، بثلاث قيم ممكنة:

```
zatca_status = rejected   -> WHERE zatca_status='rejected'
zatca_status = sent       -> WHERE zatca_status<>'rejected'
                              AND (zatca_status IN ('submitted','accepted')
                                   OR zatca_submitted IS NOT NULL)
zatca_status = not_sent   -> WHERE (zatca_status IS NULL
                              OR zatca_status NOT IN ('rejected','submitted','accepted'))
                              AND zatca_submitted IS NULL
```

⚠️ تنبيه مهم: إن كان هناك "مسار سريع" (fast path) لحساب العدد الإجمالي (Count)
يتجاهل بعض الفلاتر لتحسين الأداء على جداول كبيرة، يجب التأكد من تضمين فلتر
`zatca_status` ضمن شرط تفعيل هذا المسار السريع (أي اعتباره "فلتر موجود")، وإلا
سيظهر عدد إجمالي خاطئ (كل الفواتير بدل الفواتير المفلترة فقط) رغم أن الصفوف
المعروضة صحيحة.

المودال يستدعي هذه النقطة بالشكل:
```
GET /invoices?type=invoice&zatca_status=rejected&page=1&pageSize=20
```
ويستقبل: `{ ok: true, items: [...], total: N }`.

## 7. إعادة استخدام المنطق بدل التكرار

- منطق "عرض الفاتورة" (بناء رابط الطباعة) يجب استخراجه في دالة واحدة مشتركة
  تُستدعى من الجدول الرئيسي **ومن** جدول المودال، لأن صفوف المودال ليست بالضرورة
  موجودة ضمن الصفحة المعروضة حالياً في الجدول الرئيسي.
- منطق "إرسال يدوي" (نداء API الإرسال + معالجة النجاح/الفشل + عرض الرد تلقائياً)
  يُعاد استخدامه أيضاً بين الجدول الرئيسي والمودال عبر معالج نقر واحد مشترك.

## 8. ملخص تدفق الاستخدام (User Flow)

1. المستخدم يفتح شاشة الفواتير، والربط الإلكتروني مفعّل.
2. تظهر 3 بطاقات إحصائية بجانب البحث بالأرقام الصحيحة.
3. يضغط المستخدم على بطاقة "فشل الإرسال".
4. تُفتح نافذة منبثقة، تُظهر Spinner تحميل، ثم تجلب الصفحة الأولى (20 فاتورة)
   من الفواتير المرفوضة عبر API.
5. لكل فاتورة: رقمها، جوال عميلها، تاريخها ووقتها، طريقة دفعها، وأزرار
   (عرض/إرسال يدوي/الرد).
6. عند الضغط "إرسال يدوي": تجربة إعادة إرسال، ثم رسالة نجاح/فشل + عرض رد الهيئة
   تلقائياً في نافذة صغيرة أعلى المودال، ثم تحديث الصفحة الحالية من الجدول.
7. عند الضغط "الرد": نافذة صغيرة تعرض نص/JSON الرد أو سبب الرفض.
8. عند الضغط "عرض الفاتورة": تفتح نافذة طباعة الفاتورة كما في الجدول الرئيسي.
9. يمكن التنقل بين الصفحات إن كان عدد الفواتير الفاشلة أكبر من 20.
10. الإغلاق بأي من الطرق الثلاث المذكورة أعلاه.
