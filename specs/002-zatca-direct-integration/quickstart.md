# Quickstart: التحقق من الربط المباشر في الكاشير

دليل تحقق شامل (لا يحتوي كود تنفيذ). المرجع التفصيلي: [plan.md](./plan.md)، [data-model.md](./data-model.md)، [contracts/ipc-api.md](./contracts/ipc-api.md).

## المتطلبات

- MySQL يعمل وقاعدة الكاشير مهيأة، تشغيل التطبيق: `npm start` من جذر المشروع.
- **يجب ألا يكون Java أو Payara مثبتًا** على جهاز اختبار الوضع المباشر (هذا جزء من التحقق نفسه).
- بيئة `simulation` تتطلب OTP من بوابة فاتورة؛ بيئة `sandbox` تعمل بأي OTP (مثل 123456).

## السيناريو 1 (الأهم): استمرارية الوضع القديم بعد الترقية

1. جهّز قاعدة بيانات بحالة "عميل قديم": `app_settings.zatca_enabled=1` ولا وجود لجدول `zatca_settings`، مع وسيط Java يعمل على 8080 (أو محاكيه).
2. شغّل الإصدار الجديد.
3. **المتوقع**: يُنشأ `zatca_settings` بـ `integration_mode='legacy'`؛ إصدار فاتورة يرسلها عبر `localhost:8080` كما قبل الترقية تمامًا؛ لا تظهر أي مطالبة بإعادة ربط؛ المجدول الساعي يعيد إرسال غير المرسل عبر الوسيط.
4. تحقق: `SELECT integration_mode FROM zatca_settings` = 'legacy'، و`sales.zatca_status` تتحدث كما في السابق.

## السيناريو 2: ربط جهاز جديد بالوضع المباشر (simulation)

1. قاعدة نظيفة (`zatca_enabled=0`) → المتوقع `integration_mode='unlinked'`.
2. من شاشة "الربط الإلكتروني المباشر": أدخل بيانات المنشأة والعنوان الوطني → حفظ (جرّب VAT غير صالح أولًا وتأكد من رسالة "15 رقم").
3. اختر البيئة simulation → توليد CSR → أدخل OTP من بوابة فاتورة → شهادة الامتثال → تشغيل فحوص الامتثال الستة → شهادة الإنتاج.
4. **المتوقع**: الحالة تتدرج `csr_generated → compliance_issued → compliance_passed → production_ready` و`integration_mode='direct'`، وتظهر بطاقة الحالة تاريخ انتهاء الشهادة.
5. تحقق أمني: `SELECT private_key_enc FROM zatca_settings` بصيغة `iv:tag:cipher` (مشفر)، ولا يظهر أي سر في الواجهة أو الـ logs.

## السيناريو 3: إرسال المستندات

1. فاتورة مبسطة (عميل نقدي) → **المتوقع**: قبول REPORTED، QR مرحلة 2 يظهر على الطباعة، `zatca_submissions` فيه صف `status='reported'` وICV=1.
2. فاتورة ثانية → ICV=2 و`previous_invoice_hash` = hash الفاتورة الأولى.
3. فاتورة لعميل شركة برقم ضريبي وعنوان مكتمل → CLEARANCE وتُقبل.
4. عميل شركة بلا رقم ضريبي → رسالة منع عربية، لا يُستهلك ICV.
5. إشعار دائن (مرتجع) لفاتورة مرسلة → يُقبل مرتبطًا بأصله (النوع 381).
6. فاتورة بخصم + رسوم إضافية + دفع مختلط → تُقبل والمجاميع مطابقة (شبكة أمان الإجمالي لا تعترض).

## السيناريو 4: الانقطاع وإعادة المحاولة

1. افصل الإنترنت → أصدر 3 فواتير → **المتوقع**: البيع والطباعة طبيعيان، الصفوف `pending` مع `next_attempt_at`.
2. أعد الاتصال وانتظر دورة المجدول (أو "إعادة إرسال غير المرسل" يدويًا) → تُرسل الثلاث بترتيبها وICV متسلسل بلا فجوات مزدوجة.
3. أعد تشغيل التطبيق أثناء وجود معلقات → لا ازدواج (نفس uuid يعاد استخدامه لكل مستند).
4. فاتورة برفض بيانات (مثلاً عبث بنسبة الضريبة) → `rejected` مع السبب، ولا توقف بقية القائمة.

## السيناريو 5: الانتقال الطوعي legacy → direct

1. على جهاز السيناريو 1 (legacy يعمل): افتح شاشة الربط المباشر → يظهر تحذير أن الجهاز على الوضع القديم.
2. أكمل التهيئة حتى شهادة الإنتاج مع التأكيد الصريح.
3. **المتوقع**: `integration_mode='direct'`، الفواتير الجديدة تُرسل مباشرة (بلا 8080)، سجلات الفواتير القديمة تبقى كما هي، والسلسلة تبدأ ICV=1.
4. تأكد أنه بدون هذا الإجراء الصريح لا شيء يغير الوضع (أعد التشغيل عدة مرات → يبقى legacy).

## السيناريو 6: الشهادة

1. عدّل `certificate_expires_at` ليكون بعد 10 أيام → تنبيه "تنتهي خلال 10 يوم" عند الفتح.
2. اجعلها في الماضي → الإرسال يتعلق برسالة انتهاء الشهادة، البيع مستمر.

## اختبارات آلية

- `npm test -- __tests__/zatca-direct` — وحدات: كشف الوضع (router)، mapper (خصومات/شامل الضريبة/extra/إجمالي غير مطابق)، vault (تشفير/فك، مفتاح خاطئ).
